    前言¶

    一个罕见的机会¶

    2026 年 3 月 31 日，研究者 Chaofan Shou 在 Twitter 上发布了一条简短的消息：

    "Claude code source code has been leaked via a map file in their npm registry!"

    事情的经过并不复杂。Claude Code 在发布 npm 包时，构建产物中携带了 .map 文件，而这些 source map 文件包含了指向 Anthropic R2 存储桶中原始 TypeScript 源码的链接。这些链接是公开可访问的。任何人，只要知道去哪里找，就可以下载到完整的源代码。

    Anthropic 很快修复了这个问题。但在那之前，这份代码已经被多人存档。

    本书的研究材料，正是来自那份代码快照。

    为什么要写这本书¶

    我们每天都在使用 Prompt。写一段系统指令，调整几个措辞，测试效果，再改，再测试。这个过程往往凭直觉，凭经验，凭运气。

    关于 Prompt Engineering，已经有很多教程、博客和论文。它们告诉我们"要具体"、"要给例子"、"要用 chain-of-thought"。这些建议并没有错，但它们大多来自实验室环境下的受控实验，或者来自某个具体任务的优化经验。

    Claude Code 提供了一个完全不同的视角。

    这是一个生产级的、被数十万开发者每天使用的 AI 系统。它的 prompt 不是某个研究员在笔记本上写出来的，而是经过数百次迭代、A/B 测试、真实失败案例驱动的工程产物。代码里的每一条规则，几乎都有一个对应的真实失败在它诞生之前。代码里的注释，如 // @[MODEL LAUNCH]: capy v8 thoroughness counterweight (PR #24302) ，记录的是工程师们与模型行为反复博弈的痕迹。

    研究这份代码，相当于拿到了一家顶级 AI 公司内部的 prompt 工程手册。

    这种机会，极其罕见。

    这本书是什么¶

    本书是对 Claude Code 中所有 prompt 设计的系统性解析。

    具体来说，本书做三件事：

    第一，完整收录原文。 Claude Code 中所有的指令文本——系统 prompt 的每个 section、每个工具的使用说明、压缩摘要的生成规则——都以英文原文的形式完整呈现，逐字与源代码保持一致。随后附上中文译文，方便英语阅读不便的读者理解。

    第二，解析设计决策。 对每段 prompt，解释它为什么这样写，而不是那样写。不是泛泛而谈的"要具体"，而是："这里用了列举具体命令名而非抽象类别的技巧，因为模型对 git reset --hard 的识别远比对'破坏性操作'更精确。"

    第三，归纳可迁移的原则。 将所有设计模式提炼为系统性的原则，并说明如何将这些原则应用到你自己的 LLM 系统设计中。

    本书不是一本入门级的 Prompt Engineering 教程。如果你对 LLM 的基本工作原理、token 和 context window 的概念、tool calling 的机制还不熟悉，建议先阅读相关基础材料。

    关于源材料的说明¶

    本书所有分析基于 2026 年 3 月 31 日的 Claude Code 源码快照，对应当时 npm 发布的版本。此后 Anthropic 可能对 prompt 进行了调整，本书内容与最新版本可能存在差异。

    关于英文原文的准确性：本书收录的所有英文 prompt，均直接从源代码中提取，未做任何修改。每章写作完成后，均经过对照源代码的逐字校验。如发现任何出入，请以源代码为准。

    关于版权：Claude Code 的原始代码版权归 Anthropic 所有。本书对这些内容的引用，目的是教育和研究，属于合理使用范畴。

    本书的结构¶

    全书分为七章和两个附录，按照"先理解系统，再深入细节，最后归纳原则"的顺序组织：

    第一章：Claude Code 系统概览
    从宏观视角理解 Claude Code 是什么、由哪些部分组成，以及 prompt 在整个系统中处于什么位置。帮助读者在进入细节之前，先建立整体的系统观。

    第二章：Prompt 的组织架构
    Claude Code 的 system prompt 不是一段固定文字，而是一个动态组装系统。本章解释这个系统的设计——section 机制、工具级 prompt 的范式、静态与动态内容的分界。

    第三章：Prompt 的工程机制
    从工程实现的角度，解释 prompt 是如何被构建、缓存、按用户类型差异化，以及如何随 feature flag 变换形态的。这是理解所有成本控制和性能优化决策的基础。

    第四章：主系统 Prompt 全解析
    逐节分析系统 prompt 的每个组成部分。这是全书篇幅最长的章节，也是核心内容所在。

    第五章：工具 Prompt 全解析
    对所有工具的 prompt 进行系统分析，从最复杂的 BashTool 到最简短的 MCPTool，按分析深度分三个档次处理。

    第六章：Compact 压缩摘要 Prompt 全解析
    单独一章分析 compact 机制，因为它涉及一个完全不同的 prompt 设计问题：如何用 prompt 驱动一个 AI 系统对自身历史对话进行高质量的压缩摘要。

    第七章：Prompt 有效性的设计原则
    将前六章中出现的所有设计模式归纳为十条系统性原则，并讨论如何将这些原则迁移到读者自己的 LLM 应用开发中。

    附录 A：英文原版 Prompt 完整收录
    所有 prompt 的英文原文，按章节顺序排列，不含分析，供快速查阅。

    附录 B：中文译文完整收录
    对应附录 A 的中文版本。

    如何阅读这本书¶

    如果你想系统学习：按章节顺序阅读。每章都建立在前一章的基础上，第一章到第三章是理解后续内容的基础。

    如果你想快速查阅某个 prompt：直接跳到附录 A 或附录 B，定位到对应 section，再返回正文阅读分析。

    如果你只想了解设计原则：可以直接阅读第七章，再按需回溯前面章节中的具体案例。

    关于英文 prompt 的阅读建议：即使英语不是你的母语，也建议尽量阅读英文原文。Prompt 的效果高度依赖于具体的措辞选择——"NEVER"和"avoid"在 prompt 中传达的强度是不同的，"CRITICAL"放在句首和放在句中的效果也不同。这些细节在翻译中难以完整保留。中文译文是为了帮助理解大意，而不是代替原文。

    一点个人的观察¶

    研究这份代码的过程中，有一件事让我印象深刻。

    代码里有很多注释，记录着 prompt 修改的原因和时机。比如某个规则旁边写着：// @[MODEL LAUNCH]: False-claims mitigation for Capybara v8 (29-30% FC rate vs v4's 16.7%)。意思是，在 Capybara v8 模型上，虚假声明的发生率从 v4 的 16.7% 上升到了 29-30%，于是他们专门加了这条规则来压制。

    这说明什么？这说明 Anthropic 的工程师知道自己模型的每一个具体缺陷，并且用 prompt 来对每个缺陷打补丁。

    这不是"玄学"，这是工程。他们有 A/B 测试系统，有精确的失败率数据，有从数据到 prompt 改动再到效果验证的完整闭环。

    这让我想到，我们大多数人在写 prompt 时，其实是在用"感觉"解决一个本质上是工程问题的事情。我们没有量化的失败率数据，没有对照实验，没有跨模型版本的追踪。

    本书能提供的，是一份来自工业实践的参考样本。至于如何将这些方法应用到你自己的工作中，需要你结合自己的场景去思考。

    希望这本书对你有用。
